2. помогает не думать о навигациях, достаточно просто знать, что объект связан с другим объектам (если мы говорим именно про написание кода, а не DDL)

чем мешает:
1. Лёгкая соблазна делать public set или ICollection<T> для всех навигаций → нарушает инварианты.
2. Использование DbContext прямо в домене → смешивает персистенцию с поведением.
3. Shadow-tables или автоматические M2M могут скрывать join-entity → потеря бизнес-логики.
4. Lazy-loading → может делать агрегаты частично загруженными → инварианты не соблюдаются.

Инвариант — это правило или ограничение, которое должно всегда выполняться для объекта или агрегата во время его жизненного цикла.
В DDD инварианты гарантируют целостность доменной модели.
Инварианты — это правила целостности агрегатов

| Тип                 | Пример                                                    |
| ------------------- | --------------------------------------------------------- |
| Простое ограничение | У пользователя email должен быть уникальным               |
| Агрегатное          | У посетителя может быть только **один основной зал**      |
| Сложное правило     | Сумма платежей по заказу ≤ кредитного лимита пользователя |


Ключ: инвариант должен выполняться до и после каждой операции над агрегатом

Как Ef мешает:
1. Lazy Loading позволяяет не целиком загрузить все, что нужно для инварианта
2. Public Setters на коллекциях или свойствах, в следствии чего любой код может делать с entity все что ему вздумается
3. Несогласованные операции на join-entity
Инвариант: "один зал может быть default только у одного посетителя". напрямую делаешь db.VisitorGymRelations.Add(...), bypassing domain methods → Нарушается инвариант, потому что EF просто вставляет строки в таблицу
4. следствие из предыдущего -> использование DbContext в обход сервиса или другой части


как этого избежать:
1. Инкапсулируй коллекции и поля: private readonly List<T> + публичный IReadOnlyCollection<T>
2. Используй методы домена для изменений: Например: visitor.AddGym(gym) вместо прямого изменения коллекции
3. Проверяй бизнес-правила внутри методов: Например, проверка на единственный default gym
4. Ограничь доступ к DbContext: Пусть домен не знает про EF напрямую, EF Core работает в репозиториях/Unit of Work
5. Используй транзакции, если агрегат span’ит несколько таблиц

При этом добиться DDD становится сложным, тк не всегда является возможным поддержать инварианты + удобство их реализации без ущерба производительности (в следствии использования постоянно eager loading и тд). поэтому надо искать баланс
