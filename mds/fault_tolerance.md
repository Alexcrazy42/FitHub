# Consistency

1. 2 phase commit

Фаза 1. Prepare

1. Координатор (Transaction Manager) отправляет всем участникам (сервисам/БД) запрос:
"Может ли вы выполнить транзакцию"
2. Участники выполняются все проверки (достаточно ли денег, есть ли товар), но не фиксируют изменения
3. Каждый участник отвечает: да/нет

Фаза 2. Commit/Rollback (фиксация или откат)
Если все участники ответили "Да":
    координатор отправляет запрос Commit
    участники фиксируют изменения в своих БД

Если хотя бы один ответил "Нет" или не ответил:
    координатор отправляет Rollback
    все участники откатывают транзакцию

Пример. Банковский перевод
Допустим, нужно перевести $100 с Счёта A на Счёт B:
1. Фаза Prepare:
Банк A: "Можешь списать $100?" → ✅ (деньги есть).
Банк B: "Можешь зачислить $100?" → ✅ (счёт открыт).

2. Фаза Commit:
Координатор отправляет COMMIT.
Банк A списывает $100
Банк B зачисляет $100.

2. Saga
Saga состоит из последовательности шагов. Если один шаг падает, запускаются компенсации (откаты в обратном порядке)

Пример: Заказ в интернет-магазине
Order Service: Создаёт заказ (Pending).
Payment Service: Списание денег.
Inventory Service: Резервирование товара.
Delivery Service: Отправка заказа.


Если на шаге 3 товара нет:
Запускается компенсация:
Отмена резерва (если был).
Возврат денег.
Отмена заказа.

2 способа реализации Saga:
1. Оркестрация (Orchestration)
    Центральный координатор (оркестратор) управляет потоком.
    Сервисы вызываются последовательно (REST/RPC) или через асинхронные события.
    Плюсы: Легче отлаживать. Меньше связности между сервисами
    Минусы: Оркестратор - потенциально место отказа
2. Хореография (Choreography)
    Нет центрального координатора, сервисы общаются через события (Kafka, RabbitMQ).
    Каждый сервис сам знает, что делать после получения события.
    Плюсы: Высокая масштабируемость. Нет единой точки отказа.
    Минусы: Сложнее отслеживать состояние. Риск циклических зависимостей


Пример потока событий: 
    OrderService - OrderCreated (публикует в Kafka)
    PaymentService (подписан) - PaymentTaken
    InventoryService (подписан) - InventoryReserved
    Если ошибка → OrderFailed → все сервисы откатываются.

Проблемы Saga:
    1. Нет изоляции: пользователь может увидеть промежуточные состояния (деньги списались, заказа нет)
    2. Сложные компенсации: не все операции можно откатить (например, отправка писем)
    3. Идемпотентность: нужно гарантировать, что компенсация не выполняется дважды

🏆 Где применяется?
Электронная коммерция (оформление заказов).
Бронирование билетов (авиа, отели).
Банковские переводы (между разными системами).
Сложные бизнес-процессы (например, onboarding клиента).

🔧 Как улучшить Saga?
Идемпотентные операции (чтобы компенсация не ломала систему).
Outbox Pattern (для надёжной доставки событий).
Saga + Event Sourcing (для полного аудита изменений).




OrderService
Задача: Управление жизненным циклом заказа

Что делает: 
    создает заказ со статусом Pending
    Хранит информацию о заказе: товары, пользвоатель, стоимость, статус
    Координирует общение между другими сервисами (если используется оркестрация)

POST /orders → Создать заказ  
GET /orders/{id} → Проверить статус  
PATCH /orders/{id}/cancel → Отменить заказ  

PaymentService
Задача: Обработка платежей

Что делает:
    проверяет платежные данные (карта, баланс кошелька)
    списывает деньги с учетной записи пользователя
    возвращает средства при отмене заказа (компенсирующая транзакция)

POST /payments/charge → Списать деньги  
POST /payments/refund → Вернуть деньги  

InventoryService (Сервис инвентаризации)
Задача: управление остатками товаров

Что делает:
    Проверяет наличие товара на складе
    Резервирует товар для заказа (чтобы его не купили дважды)
    Отменяет резерв при откате транзакции

POST /inventory/reserve → Зарезервировать товар  
POST /inventory/cancel-reserve → Отменить резерв  



DeliveryService 
Задача: Организация доставки заказа

Что делает:
    создает заявку на доставку
    отслеживает статус доставки
    отменяет доставку, если заказ аннулирован

POST /deliveries → Создать доставку  
POST /deliveries/{id}/cancel → Отменить доставку  


Взаимодействие сервисов путем Saga

Сценарий успешного заказа:
OrderService создаёт заказ (Pending).
PaymentService списывает деньги.
InventoryService резервирует товар.
DeliveryService запускает доставку.
OrderService переводит заказ в статус Completed.

Сценарий отката (если товара нет):
OrderService создаёт заказ.
PaymentService списывает деньги.
InventoryService не может зарезервировать товар → запускается компенсация:
PaymentService.refund() — возврат денег.
OrderService.cancel() — отмена заказа.

Почему именно так?
OrderService — центральный "мозг" процесса.
PaymentService и InventoryService — критичные шаги (без денег или товара заказ невозможен).
DeliveryService — финальный этап (можно отменить, даже если доставка уже создана).

Что будет, если один из сервисов упадёт?
Синхронная Saga (оркестрация):
Таймаут → откат через компенсации.
Асинхронная Saga (хореография):
Событие не обработано → повторная отправка после восстановления сервиса.
