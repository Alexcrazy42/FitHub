# Тестирование курьерной доставки без реальных курьеров

## Проблема

Система доставки завязана на живых людей: курьер принимает заказ, едет по маршруту, обновляет геопозицию, подтверждает вручение. Воспроизвести это в тесте без реального курьера на карте невозможно — если подходить в лоб.

Решение: **курьерские боты** — программные агенты, которые имитируют поведение реального курьера с точки зрения бэкенда и клиентского приложения.

---

## Как работает курьерский бот

Бот — это процесс, который от имени тестового курьера:
1. Берёт заказ из очереди (или его назначают через API)
2. Движется по маршруту — шлёт GPS-координаты с нужным интервалом
3. В нужные моменты меняет статус: `assigned → picked_up → on_the_way → delivered`
4. Имитирует финальное действие (фото, код подтверждения, подпись)

С точки зрения бэкенда и покупательского приложения — курьер настоящий. Разница только в том, что GPS-точки генерирует скрипт, а не реальный телефон.

```
Реальный курьер:
  Курьер с телефоном → GPS → Курьерское приложение → Backend → Покупатель

Бот:
  Route Script → fake GPS stream → тот же Backend endpoint → тот же Покупатель
```

## Архитектура бота

### Минимальный бот (один скрипт)

```python
import asyncio, httpx, time
from dataclasses import dataclass

@dataclass
class GpsPoint:
    lat: float
    lon: float

MOSCOW_ROUTE = [
    GpsPoint(55.7558, 37.6173),
    GpsPoint(55.7560, 37.6180),
    # ... точки маршрута
    GpsPoint(55.7600, 37.6250),
]

async def run_courier_bot(order_id: str, courier_token: str):
    async with httpx.AsyncClient() as client:
        headers = {"Authorization": f"Bearer {courier_token}"}

        # 1. Принять заказ
        await client.post(f"/orders/{order_id}/accept", headers=headers)

        # 2. Двигаться по маршруту
        for point in MOSCOW_ROUTE:
            await client.post("/courier/location", headers=headers, json={
                "lat": point.lat,
                "lon": point.lon,
                "accuracy": 5.0,
            })
            await asyncio.sleep(2)  # имитируем движение ~30 км/ч

        # 3. Забрать заказ из ресторана/склада
        await client.post(f"/orders/{order_id}/picked-up", headers=headers)

        # 4. Двигаться к покупателю
        for point in DELIVERY_ROUTE:
            await client.post("/courier/location", headers=headers, json={
                "lat": point.lat, "lon": point.lon,
            })
            await asyncio.sleep(2)

        # 5. Завершить доставку
        await client.post(f"/orders/{order_id}/delivered", headers=headers, json={
            "confirmation_code": "1234",
            # или: "photo_url": "...", "signature": "..."
        })
```

### Что можно контролировать

Бот параметризуется под любой сценарий:

```python
@dataclass
class BotScenario:
    route: List[GpsPoint]
    speed_kmh: float = 30.0
    pickup_delay_sec: int = 0       # задержка у ресторана
    gps_noise_meters: float = 0.0   # имитация неточного GPS

    offline_gap_sec: int = 0        # пропасть с радаров на N секунд
```

## Сценарии, которые становятся тестируемыми

### Счастливый путь
- Заказ создан → бот принял → доехал → доставил → статус у покупателя обновился

### Отмена курьером на разных стадиях
- Бот принял заказ, но до pickup отменил → система должна переназначить другому
- Бот забрал заказ и отменил в дороге → отдельная логика (товар уже у курьера)

### Проблемы с GPS
- Бот шлёт точки с большим jitter → карта у покупателя не дёргается
- Бот пропадает с радаров на 2 минуты → покупатель видит «связь с курьером потеряна»
- Бот шлёт координаты за пределами разумного (телепортация) → система фильтрует аномалии

### Опоздание
- Бот движется с нулевой скоростью → система должна обновить ETA и уведомить покупателя
- Бот превысил расчётное время доставки → эскалация в поддержку / компенсация

### Провал доставки
- Бот сигнализирует «не нашёл адрес» → покупатель получает уведомление
- Бот сигнализирует «покупатель не открыл дверь» → логика повторного визита или возврата

### Нагрузочный тест
- 50 ботов одновременно ведут заказы → проверяем что WebSocket/SSE не падает, база не перегревается

## Генерация маршрутов

Чтобы боты двигались правдоподобно, нужны реальные маршруты — не прямая линия от A до B.

**Вариант 1: OSRM (Open Source Routing Machine)**
```python
# Запрашиваем реальный маршрут по дорогам
async def get_route(origin: GpsPoint, destination: GpsPoint) -> List[GpsPoint]:
    url = f"http://router.project-osrm.org/route/v1/driving/{origin.lon},{origin.lat};{destination.lon},{destination.lat}"
    response = await client.get(url, params={"geometries": "geojson", "overview": "full"})
    coordinates = response.json()["routes"][0]["geometry"]["coordinates"]
    return [GpsPoint(lon=c[0], lat=c[1]) for c in coordinates]
```

OSRM можно поднять локально через Docker — не нужно зависеть от внешнего API.

**Вариант 2: Заранее записанные маршруты**
Один раз проехать реальные маршруты с записью GPS-трека (GPX-файл), потом воспроизводить их в тестах. Самый реалистичный вариант: боты повторяют реальные треки реальных курьеров.

## Инфраструктура для запуска ботов

### Для ручного QA (один бот)
Скрипт с CLI: `python courier_bot.py --order-id=123 --scenario=happy_path`

### Для регрессионных тестов (несколько ботов)
Боты запускаются как часть интеграционного теста, в параллельных задачах:

```python
async def test_simultaneous_deliveries():
    orders = [await create_test_order() for _ in range(5)]
    bots = [run_courier_bot(order.id, scenario=HappyPath()) for order in orders]
    results = await asyncio.gather(*bots)
    assert all(r.status == "delivered" for r in results)
```

### Для нагрузочного тестирования
k6 или Locust с кастомным scenario, где каждый виртуальный пользователь — это курьерский бот.

## Ограничения подхода

**Что боты НЕ тестируют:**
- Реальное мобильное приложение курьера (UI, UX, push-уведомления на телефоне)
- Производительность GPS на реальном железе (батарея, точность в зданиях)
- Поведение курьера как человека (звонок покупателю, фото у двери)
- Сетевые условия реального телефона (LTE → 3G → нет связи)

**Для этого нужен следующий уровень** — тестирование с реальными телефонами в staging-среде (небольшая группа внутренних тестировщиков, играющих роль курьеров).

## Связь с другими подходами

| Подход | Роль в тестировании доставки |
|---|---|
| **Курьерский бот** | Тестирует бэкенд-логику и клиентский опыт покупателя |
| **Contract Testing** | Фиксирует формат событий от курьерского приложения — если оно стороннее |
| **Synthetic Monitoring** | Гоняет бота в продакшне с тестовым заказом, проверяет что флоу жив |
| **Chaos Engineering** | Бот внезапно пропадает, бот шлёт невалидные координаты — как реагирует система |
