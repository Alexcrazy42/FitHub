# Тестирование: сложные сценарии воспроизведения

## Проблема

Большинство тестов предполагают детерминированную среду: вызвал метод — получил ответ. Но реальные системы работают иначе. Внешний платёжный провайдер присылает вебхук через 5–30 секунд. Курьерская служба подтверждает доставку асинхронно. RabbitMQ доставляет сообщение «когда-нибудь». Всё это делает обычный подход «запустил тест — проверил результат» нерабочим.

Эта лекция о том, как индустрия решает пять классов таких проблем.

---

## 1. Contract Testing

### Суть проблемы

У тебя есть сервис A, который вызывает сервис B (или B присылает вебхук в A). Ты пишешь интеграционный тест — поднимаешь мок B, проверяешь, что A работает корректно. Тест зелёный. Потом команда B молча меняет поле в JSON-ответе — например, переименовывает `user_id` в `userId`. Твой мок не знает об этом. Тест всё ещё зелёный. В продакшне — сломано.

Это классическая проблема расхождения контракта: ты тестируешь не реальный B, а своё представление о нём.

### Решение: Contract Testing (контрактное тестирование)

Идея: формализовать «контракт» между двумя сторонами и тестировать соответствие этому контракту независимо на каждой стороне.

**Потребитель (Consumer)** описывает, что именно он ожидает от провайдера: какие поля нужны, какого типа, какие статус-коды. Эти ожидания фиксируются в файле-контракте (обычно JSON/YAML).

**Провайдер (Provider)** запускает тесты, которые проверяют: «Удовлетворяю ли я всем контрактам, которые от меня ожидают?»

Обе стороны тестируются независимо и изолированно. Синхронизация происходит через файл контракта, а не через живой сервис.

### Как это работает на практике (Pact)

Самый популярный инструмент — [Pact](https://pact.io/).

**На стороне потребителя (например, ваш фронтенд или другой сервис):**
```typescript
// Потребитель описывает взаимодействие
await provider.addInteraction({
  state: 'order 123 exists',
  uponReceiving: 'a request for order status',
  withRequest: {
    method: 'GET',
    path: '/orders/123',
  },
  willRespondWith: {
    status: 200,
    body: {
      id: 123,
      status: 'delivered', // потребитель говорит: мне нужно именно это поле
    },
  },
});
```

Pact запускает локальный мок-сервер, прогоняет тест потребителя против него, и сохраняет файл контракта `consumer-provider.json`.

**На стороне провайдера:**
```csharp
// Провайдер загружает контракт и верифицирует его против реального кода
[Fact]
public async Task VerifyPactWithOrderConsumer()
{
    var config = new PactVerifierConfig();
    new PactVerifier(config)
        .ServiceProvider("OrderService", new Uri("http://localhost:5000"))
        .HonoursPactWith("FrontendApp")
        .PactUri("./pacts/frontendapp-orderservice.json")
        .Verify();
}
```

### Применительно к вебхукам

Вебхуки — это «обратный» контракт: провайдер (например, платёжная система) является отправителем, а твой сервис — потребителем. Contract Testing здесь работает так:

- Ты описываешь, какой формат вебхука ожидаешь (поля, типы, сигнатуру HMAC).
- Тест симулирует входящий вебхук с этим payload и проверяет, что твой обработчик корректно реагирует.
- Отдельно — провайдер (или ты с его документацией) верифицирует, что реальный вебхук соответствует задокументированному контракту.

Вместо того чтобы ждать реального вебхука в тесте, ты тестируешь соответствие контракту. Ждать ничего не нужно.

### Когда применять

- Интеграции с внешними API (платёжки, SMS-шлюзы, CRM)
- Вебхуки от сторонних систем
- Микросервисная архитектура — особенно когда команды работают независимо
- Любая граница между системами, где изменение одной стороны может сломать другую

## 2. Synthetic Monitoring

### Суть проблемы

Некоторые сценарии вообще невозможно нормально воспроизвести в тестовой среде:

- Курьер подтверждает доставку с мобильного приложения — как это эмулировать?
- Пользователь стоит в очереди в реальном магазине и оплачивает через POS-терминал.
- Провайдер рассылает события раз в сутки по расписанию на своей стороне.
- Биржевой API возвращает данные, которые зависят от реального рынка.

Мок здесь не поможет: ты тестируешь не интеграцию, ты тестируешь своё представление о ней. А реальная среда — непредсказуема.

### Что такое Synthetic Monitoring

Synthetic Monitoring — это **не тестирование**, это **наблюдение**. Вместо того чтобы запускать тесты в тестовой среде, ты запускаешь **синтетические транзакции прямо в продакшне** и наблюдаешь за результатом.

«Синтетическая транзакция» — это заранее заскриптованное, предсказуемое действие, которое система выполняет автоматически. Не реальный пользователь, а робот, который имитирует реального пользователя.

**Примеры:**
- Каждые 5 минут бот делает тестовый заказ на специальный тестовый товар (который не влияет на склад), и система проверяет, что весь флоу прошёл.
- Каждый час отправляется тестовый платёж через реальный платёжный шлюз в sandbox-режиме с мониторингом латентности.
- Бот проходит авторизацию и проверяет, что критические страницы открываются за < 2 секунд.

### Synthetic vs Real User Monitoring (RUM)

| | Synthetic Monitoring | Real User Monitoring (RUM) |
|---|---|---|
| Источник данных | Заскриптованный бот | Реальные пользователи |
| Когда работает | 24/7, независимо от трафика | Только когда есть пользователи |
| Воспроизводимость | Детерминирован (одни и те же шаги) | Зависит от поведения пользователей |
| Что обнаруживает | Регрессии, деградацию производительности | Реальный UX, edge cases пользователей |
| Алерты | Можно настроить на конкретные пороги | Статистические аномалии |

### Это не тест — это зонд

Ключевое отличие от тестирования: synthetic monitoring не блокирует деплой и не «проваливается». Он непрерывно работает в продакшне и сигнализирует, когда что-то пошло не так. Это как кардиомонитор — он не лечит, он наблюдает.

Отвечая на твой вопрос: нет, это не постепенная раскатка через % пользователей — это другая концепция (canary deployment / feature flags). Synthetic Monitoring — это именно наблюдение: живёт отдельно от деплоя, работает всегда, использует реальную инфраструктуру, но генерирует контролируемую нагрузку.

### Инструменты

- **Datadog Synthetic Tests** — самый популярный enterprise-инструмент
- **Checkly** — ориентирован на разработчиков, тесты пишутся как Playwright/Puppeteer скрипты
- **Grafana k6 + Cloud** — open-source, можно запускать из CI или как scheduled job
- **AWS CloudWatch Synthetics** — если инфраструктура на AWS

### Применительно к курьерам и логистике

Для систем с физическим миром (логистика, IoT, курьеры) synthetic monitoring — часто единственный реалистичный вариант:

- Создаётся тестовый заказ с тестовым адресом
- Курьерская система обрабатывает его в специальном «sandbox» режиме (если провайдер его поддерживает)
- Проверяется, что статусы приходят корректно и в разумное время
- Тестовый заказ автоматически отменяется/закрывается

Если sandbox нет — иногда единственный вариант это e2e smoke test с реальным недорогим заказом раз в сутки.

## 3. Chaos Engineering

### Суть проблемы

Ты написал хорошие unit-тесты. Написал интеграционные тесты. Всё зелёное. Деплоишь — и через неделю в продакшне:

- Один из трёх инстансов приложения упал, остальные не справляются с нагрузкой
- Сеть между сервисами начала давать 5% потерь пакетов
- PostgreSQL замедлился из-за неоптимального autovacuum
- RabbitMQ перезагрузили — сообщения в полёте потеряны
- S3 вернул 503 в самый неподходящий момент

Никакой обычный тест это не воспроизведёт, потому что в тестовой среде всё идеально: сеть не теряет пакеты, сервисы не падают, ресурсов хватает.

### Что такое Chaos Engineering

Chaos Engineering — это **намеренное внесение отказов** в систему (в том числе в продакшн), чтобы проверить, как она себя ведёт, и устранить уязвимости до того, как они проявятся сами.

Термин popularized Netflix, которые буквально убивали случайные EC2-инстансы в продакшне с помощью инструмента «Chaos Monkey».

### Принципы

1. **Hypothesis-driven**: сначала формулируешь гипотезу — «если один инстанс упадёт, система продолжит работать с деградацией не более 10%».
2. **Blast radius control**: начинаешь с минимального воздействия (1% трафика, один инстанс), постепенно расширяешь.
3. **Автоматический rollback**: если метрики выходят за пределы acceptable — хаос останавливается автоматически.
4. **Production first** (в зрелых командах): staging-среда не отражает реальное поведение под нагрузкой. Настоящий chaos engineering идёт в продакшн.

### Типичные эксперименты

| Отказ | Что проверяем |
|---|---|
| Убиваем случайный pod/инстанс | Есть ли graceful degradation, работает ли reconnect |
| Добавляем latency 500ms в сеть | Таймауты настроены корректно? UI сообщает пользователю? |
| Заполняем диск на 95% | Приложение gracefully отказывает или падает? |
| Отключаем внешний API (платёжку) | Retry/fallback работает? Очередь не переполняется? |
| Убиваем RabbitMQ на 30 секунд | Сообщения не теряются? Publisher подтверждения работают? |
| CPU spike на БД | Connection pool исчерпывается или корректно очередится? |

### Для FitHub-сценариев

В контексте FitHub (SignalR, RabbitMQ, S3, PostgreSQL):

**Эксперимент: S3 недоступен во время загрузки файла**
- Гипотеза: пользователь получит понятную ошибку, presigned URL не будет сохранён в БД, cleanup job не будет пытаться удалить несуществующий файл.
- Инструмент: toxiproxy (добавить latency/disconnect на S3 endpoint).

**Эксперимент: RabbitMQ перезагружается во время отправки сообщения**
- Гипотеза: сообщение либо будет доставлено (если publisher confirm получен до рестарта), либо повторно отправлено (если нет) — дублей нет благодаря idempotency key.

**Эксперимент: SignalR hub pod убивается**
- Гипотеза: клиент переподключится через другой pod, используя backplane (Redis/RabbitMQ), и не потеряет контекст сессии.

### Инструменты

- **Chaos Monkey** (Netflix) — убивает случайные инстансы
- **Toxiproxy** (Shopify) — прокси для симуляции сетевых проблем (latency, disconnect, bandwidth limit). Очень удобен для локальной разработки и CI.
- **Chaos Mesh / LitmusChaos** — Kubernetes-native chaos
- **AWS Fault Injection Service (FIS)** — managed chaos для AWS
- **Gremlin** — enterprise SaaS-платформа для chaos

### Toxiproxy пример (для интеграционных тестов)

```csharp
// В TestApplication можно добавить toxiproxy между тестом и S3
var proxy = await toxiproxyClient.CreateProxyAsync("s3", "localhost:9001", "minio:9000");

// Инжектируем latency
await proxy.AddToxicAsync(new LatencyToxic { Latency = 3000, Jitter = 500 });

// Запускаем операцию загрузки
var result = await fileService.UploadAsync(stream);

// Убеждаемся что таймаут обработан корректно
result.IsSuccess.ShouldBeFalse();
result.Error.ShouldBe(FileUploadError.Timeout);

// Убираем токсин — система должна восстановиться
await proxy.RemoveToxicAsync("latency_downstream");
```

## 4. Canary Deployment и раскатка на процент пользователей

### Суть

Canary deployment — это стратегия деплоя, при которой новая версия кода сначала получает только малый процент трафика (например, 1–5%), пока основная масса пользователей работает со старой версией. Если метрики в порядке — процент увеличивается. Если что-то пошло не так — откат.

Называется «canary» по аналогии с канарейками, которых шахтёры брали в шахту: если птица умирала от газа — люди успевали выбраться.

### Как это работает технически

```
         ┌─────────────────────────────────┐
         │         Load Balancer            │
         └──────────┬──────────────┬────────┘
                    │ 95%          │ 5%
         ┌──────────▼──────┐  ┌───▼──────────────┐
         │  Stable v1.2.3  │  │  Canary v1.3.0   │
         │  (old version)  │  │  (new version)   │
         └─────────────────┘  └──────────────────┘
```

Дополнительно часто используется **feature flags** (LaunchDarkly, Unleash, Flagsmith) — тогда один и тот же деплой, но новая фича включена только для выбранных пользователей, групп, или % аудитории.

### Убирает ли это регрессии полностью?

**Нет. Категорически нет.** Вот почему:

**1. Раскатка не покрывает редкие пути**
Если баг воспроизводится в 0.01% случаев (например, при конкретной комбинации данных пользователя), то при 5% canary он может просто не встретиться на этом этапе. Ты поднимешь до 100% — и он выстрелит.

**2. Некоторые баги проявляются только при полной нагрузке**
Race condition в connection pool, проблема с autovacuum под нагрузкой — при 5% трафика симптом может быть ниже порога обнаружения.

**3. Необратимые операции**
Если новая версия пишет данные в новом формате, который старая версия не умеет читать — у тебя 5% пользователей уже «испорчены». Откат не исправит их данные.

**4. Нет покрытия новых флоу**
Если новая фича ещё не использовалась никем из 5% canary-пользователей — у тебя ноль данных о ней.

**5. Бизнес-критичные события**
Если у тебя 1 платёж в сутки от VIP-клиента, а баг именно в платёжном флоу — ты можешь сломать этого клиента ещё до того, как canary успел что-то показать.

### Что canary действительно даёт

- **Ранний сигнал о деградации**: ошибки, p99 latency, crash rate — видно сразу.
- **Ограниченный blast radius**: если что-то пошло не так, затронуто 5%, а не 100%.
- **Обнаружение проблем инфраструктуры**: новая версия течёт по памяти, не справляется с нагрузкой — видно по метрикам инстанса.

### Правильный подход

Canary — это **последний эшелон защиты**, а не замена тестирования:

```
Unit tests → Integration tests → Contract tests → Staging → Canary → Full rollout
```

Каждый слой ловит свой класс проблем. Canary ловит то, что не удалось воспроизвести в предыдущих слоях — реальную нагрузку, реальные данные, реальную инфраструктуру.

## 5. Моки и стабы внешних сервисов

### Проблема, которую ты описал

Реальная ситуация: интегрируешься с внешней системой (банк, курьерская служба, страховой брокер). Система присылает вебхук или событие в RabbitMQ — но только когда реальный бизнес-процесс завершён на их стороне. Иногда это 2 недели ожидания одобрения заявки. Иногда — зависит от действий живого человека. Воспроизвести в тесте невозможно.

Решение — **event simulation**: в интерфейсе (adminке, Swagger, Postman, отдельном UI) есть кнопка «Сгенерировать событие X для заявки Y». Нажимаешь — и система ведёт себя так, будто это событие реально пришло от внешней системы.

### Как это правильно реализовать

Ключевой принцип: код обработчика события не должен знать, откуда оно пришло — из RabbitMQ от реального провайдера или из adminки.

```
Реальный путь:
  RabbitMQ Consumer → IEventHandler<ApplicationApprovedEvent> → бизнес-логика

Симулированный путь:
  Admin API → IEventHandler<ApplicationApprovedEvent> → та же бизнес-логика
```

Обе ветки сходятся в одну точку входа. Это важно: если симуляция вызывает отдельный метод в обход обработчика — ты тестируешь не то, что будет в продакшне.

```csharp
// Один обработчик для обоих путей
public class ApplicationApprovedEventHandler : IEventHandler<ApplicationApprovedEvent>
{
    public async Task Handle(ApplicationApprovedEvent @event)
    {
        // реальная бизнес-логика
    }
}

// RabbitMQ consumer — просто транспортный слой
public class ApplicationApprovedConsumer : IConsumer<RabbitMqApplicationApprovedMessage>
{
    private readonly IEventHandler<ApplicationApprovedEvent> _handler;

    public async Task Consume(ConsumeContext<RabbitMqApplicationApprovedMessage> context)
    {
        var @event = context.Message.ToEvent(); // маппинг
        await _handler.Handle(@event);
    }
}

// Admin endpoint — тот же обработчик
[HttpPost("admin/simulate/application-approved")]
[Authorize(Roles = "Admin")]
public async Task SimulateApproval([FromBody] SimulateApprovalRequest request)
{
    var @event = new ApplicationApprovedEvent(request.ApplicationId, request.ApprovedAt);
    await _handler.Handle(@event);
}
```

### Насколько хорош этот подход

**Сильные стороны:**

- **Скорость регресса**: вместо 2 недель ожидания — нажал кнопку, получил результат за секунды.
- **Воспроизводимость**: любой разработчик, тестировщик или QA может воспроизвести любой сценарий в любой момент.
- **Edge cases**: можно симулировать редкие события — «отклонение с причиной X», «таймаут», «частичное одобрение» — которые в реальности встречаются раз в год.
- **Параллельность**: несколько тестировщиков могут работать одновременно, не конкурируя за «живые» события.
- **Автоматизация**: эти же эндпоинты можно вызывать из автотестов.

**Слабые стороны и риски:**

**1. Расхождение с реальным форматом события**
Реальный провайдер может присылать поля, о которых ты не знаешь. Или менять формат без предупреждения. Симулятор будет отставать. Это решается contract testing (см. секцию 1) — контракт фиксирует реальный формат.

**2. Симуляция не проверяет транспортный слой**
Если у тебя ошибка в RabbitMQ consumer (десериализация, dead letter, acknowledgment) — симуляция через admin API её не поймает. Транспортный слой нужно тестировать отдельно.

**3. Соблазн упростить симулятор**
Если симулятор принимает `applicationId` и сам заполняет «достаточные» поля — ты не тестируешь обработку реального payload. Симулятор должен принимать максимально близкую к реальной структуру события, а не упрощённую.

**4. Безопасность**
Эти эндпоинты должны быть доступны только в dev/staging, или за строгой авторизацией. Если симулятор «одобряет заявки» — в продакшне это дыра.

### Когда это имеет смысл — и почему да

Этот подход **абсолютно имеет смысл** как часть правильно выстроенного тестирования. Его используют крупные компании под разными названиями: test harness, event simulator, stub provider.

Полная картина для сценария «внешний сервис присылает событие»:

| Слой | Что тестируется | Инструмент |
|---|---|---|
| Unit test | Бизнес-логика обработчика | Обычный xUnit, без RabbitMQ |
| Contract test | Формат события соответствует контракту провайдера | Pact / документация провайдера |
| Integration test | RabbitMQ consumer корректно десериализует и вызывает обработчик | TestContainers + реальный RabbitMQ |
| Регресс / QA | Полный флоу от события до результата | Admin симулятор → проверка в UI/БД |
| Synthetic monitoring | Флоу работает прямо сейчас в продакшне | Scheduled job с тестовой заявкой |

Симулятор закрывает именно «регресс / QA» строку — и это ровно то место, где ему быть.

## 6. Тестирование физического мира: симуляторы железа

### Три уровня симуляции физического мира

Индустрия выработала несколько подходов в зависимости от того, насколько «реальным» нужно быть тестирование.

#### Уровень 1: Software Simulation (чисто программный)

Никакого железа. Симулятор — это сервис или скрипт, который генерирует те же события, что генерировало бы реальное устройство.

```python
# Симулятор телематики: гоняет виртуальную машину по маршруту
async def simulate_trip(car_id: str, route: List[GpsPoint]):
    await send_event(car_id, "ignition_on")
    for point in route:
        await send_event(car_id, "gps_update", lat=point.lat, lon=point.lon, speed=60)
        await asyncio.sleep(1)
    await send_event(car_id, "ignition_off")
    await send_event(car_id, "trip_completed", distance_km=12.4)
```

Плюсы: дёшево, запускается в CI, можно параллелить сколько угодно «машин».
Минусы: не тестирует реальный протокол и реальное железо. Если TCU присылает данные в неожиданном порядке из-за особенностей прошивки — симулятор этого не воспроизведёт.

#### Уровень 2: Hardware-in-the-Loop (HIL)

Реальное физическое устройство (или точная копия платы TCU) подключается к тестовой инфраструктуре. Устройство запускает настоящую прошивку и настоящий протокол, но вместо реальной машины получает сигналы от симулятора CAN-шины.

```
[CAN Bus Simulator] ──сигналы──→ [Реальный TCU / его копия] ──MQTT──→ [Backend]
   (ПО на ПК, генерирует                 (настоящее железо,               (тот же продакшн
    сигналы: скорость,                    настоящая прошивка)              бэкенд или staging)
    обороты, GPS, замки)
```

Это либо настоящий TCU, либо dev-версия платы с такой же прошивкой, только без корпуса автомобиля вокруг.

**Почему HIL лучше чисто программного симулятора:**
- Тестирует настоящую прошивку с её реальными багами и особенностями
- Тестирует реальный сетевой протокол (MQTT, CoAP, проприетарный)
- Находит проблемы буферизации, порядка событий, таймаутов соединения

#### Уровень 3: Digital Twin (цифровой двойник)

Полноценная программная модель физического объекта, которая в реальном времени синхронизируется с реальным объектом (или заменяет его для тестирования). Используется в IoT, производстве, авиации, строительстве.

Это уже выходит за пределы тестирования ПО и переходит в область моделирования бизнес-процессов.

### Другие домены — те же принципы

Принцип «коробочки» применяется везде, где есть физический мир:

| Домен | Что симулируют | Как |
|---|---|---|
| Каршеринг / телематика | Автомобиль, GPS, CAN-шина | HIL: реальный TCU + CAN симулятор |
| Умный дом (IoT) | Датчики температуры, замки, камеры | Программный эмулятор устройства + MQTT broker |
| Платёжные терминалы | POS-терминал, NFC-считыватель | Сертифицированный программный симулятор (Visa/MC требуют) |
| Логистика / курьеры | Курьер с телефоном на маршруте | Скрипт, гоняющий GPS-точки по реальному маршруту |
| Медицинские устройства | Кардиомонитор, инсулиновая помпа | Физический HIL — требования FDA/CE |
| Банковские карты | EMV-чип, пин-пад | Тестовые карты + симулятор эквайринга |

### Когда какой уровень выбирать

**Software simulation** — для CI/CD, нагрузочного тестирования, тестирования бизнес-логики бэкенда. Дёшево, быстро, масштабируется.

**HIL (коробочка)** — когда важно протестировать реальный протокол, прошивку, или когда мобильный разработчик тестирует полный флоу «приложение ↔ железо». Нужно физическое устройство, но не нужна вся машина.

**Реальная среда с ограниченным доступом** — staging-парковка с 2-3 реальными машинами под тестовыми аккаунтами. Для финального QA перед релизом. Дорого, не масштабируется, но иногда необходимо.

### Как это соотносится с остальными подходами

Симулятор железа решает проблему на уровне интеграционного и QA-тестирования. Но он не отменяет остального:

- **Contract testing**: формат сообщений от TCU до бэкенда всё равно нужно зафиксировать — прошивка обновляется независимо от приложения.
- **Synthetic monitoring**: после деплоя нужно убедиться что реальные машины в продакшне всё ещё отправляют данные корректно.
- **Chaos Engineering**: что будет если TCU потеряет связь на 5 минут в середине поездки? Это стоит проверить намеренно на HIL-стенде, а не ждать пока это случится в продакшне.

## Итого: когда что применять

| Проблема | Подход |
|---|---|
| Разные команды меняют API независимо, контракт расходится | Contract Testing |
| Вебхук от внешней системы — не можем его инициировать в тесте | Contract Testing (тестируем обработчик против задокументированного контракта) |
| Физический мир (курьеры, IoT) — не воспроизводимо | Synthetic Monitoring |
| Нужно убедиться что критический флоу работает прямо сейчас | Synthetic Monitoring |
| Хочу знать что система деградирует gracefully, а не падает целиком | Chaos Engineering |
| Таймауты, ретраи, circuit breaker — проверить что они реально работают | Chaos Engineering |
| Новый код в продакшн с минимальным blast radius | Canary Deployment |
| Нужен ранний сигнал о деградации под реальной нагрузкой | Canary Deployment |
| Регресс на сценарий с вебхуком без ожидания 2 недели | Admin event simulator (мок внешнего сервиса) |
| Edge case «отклонение заявки с редкой причиной» — воспроизвести вручную | Admin event simulator |
| Нужно протестировать мобильное приложение против реального железа без реального объекта | HIL-симулятор (коробочка) |
| Тестирование бизнес-логики бэкенда с физическими событиями в CI | Software simulation (скрипт-эмулятор устройства) |
| Финальный QA перед релизом в физическом домене | Staging-среда с реальными объектами (ограниченный доступ) |

Все подходы дополняют, а не заменяют обычное тестирование. Unit и интеграционные тесты проверяют логику. Contract testing — контракты на границах. Synthetic monitoring — работоспособность в реальном времени. Chaos engineering — устойчивость к отказам.
